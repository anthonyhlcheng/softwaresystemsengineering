Software Testing

Software Testing is an essential part of the software development process as it produces software bugs, defects or problems that can be fixed before the product is released onto the market.

Overview History Test Levels Processes Automation Research People References

Overview

Testing can help developers realise whether their program meets the requirements set by the stakeholders; if it works as intended; if it can be installed in their environments without error; and if it meets certain standards.1 It is absolutely necessary to carry out testing to ensure the quality of the product has not suffered and that the product will not fail, as this could be very costly to the company in the future.

It must be noted that testing cannot identify all defects or bugs in a program. Instead, it compares the behaviour and output of the product with oracles ("a mechanism for determining whether a test has passed or failed").2 This could be done with exact or approximate results; documentation; statistical results or even manual analysis - i.e. a human checks whether the program works well).

History

From the start of software development up until 1956, testing and debugging a program were the same thing.
From 1957 to 1978, the two started to separate with testing was more to do with whether the program satisfied the requirements. This was known as the Demonstration oriented period.
In 1979, it was formalised by Glenford J. Myers that the two should be clearly defined as separate processes. This led onto many in the software development community wanting to break down what was then debugging, even further into smaller subtopics.
From 1979 to 1982, began the destruction phase, where the whole point of testing was to find errors.
From 1983-1987, the evaluation phase began where it started to focus more on quality and evaluation of the product.
From 1988, the prevention phase was introduced where testing had to ensure that defects were found and prevented and the overall product met its requirements.3

Testing Levels

Test levels are a hierarchy that uses different kinds of tests as the project progresses. There are four different levels that are part of the standard. Unit Testing; Integration Testing System Testing and Acceptance Testing. As shown in the diagram below, the different tests are designed for more complex items as you go further into developing a project.



Unit Testing:
Unit testing aims to verify each component of the software by separating it from the rest of the code and performing tests that show that it is in full working order.

Integration Testing:
Integration testing aims to test multiple components, to see if they correctly work together.

System Testing:
System testing tests all components within the software to ensure that it means specifications that the software must have. It is important to get this stage of the testing process correct as this is almost the finished product that the end users will expect to see.

Acceptance Testing:
This is where the product will be tested to a point where testers can say that the product can go ahead and be deployed. The tests can range anywhere from finding grammatical errors to finding major security errors.4

Testing Process

The kind of development model chosen by management has a great effect on how testing is carried out.

Waterfall Development Model
It is very common practice to wait for the whole product to be developed before testing is carried out. Developers have no idea how good the software works until it nears the very end of the whole process. If it fails, it has to go back to the development and the process repeated.

Agile or Extreme Programming
Within these two processes, unit tests are coded first and tests done on components. It is expected to fail and as this is done again and again, the component will become better over time. As more components are built, the next level of testing is introduced whilst unit testing is still in progress for the rest of the project lifecycle. The goal is to increase the amount of testing effort done and do testing more often.

Top Down and Bottom Up
Top down is when testers test the highest level of components / systems and work their way down towards smaller components. Bottom up is the opposite. This is only helpful when the whole system has been developed and helps the reporting of testing progress in a quantitive form.5

Automation & Tools

More and more testing involves some sort of automation and there are tools out there to help. Although automation cannot completely reproduce results that manual testing can achieve, it can be useful for regression testing or very repetitive testing tasks.

Tools:

Program Monitors monitor part of or all of the code and allows testers complete control of the execution of the code and reports back.
Formatted dump or symbolic debugging allows testers to view errors at certain points of a program.
Automated Functional GUI testing tools repeat certain tests through the GUI.
Benchmarks allow testers to view performance data that could be essential to see if it meets time requirements etc.
Performance analysis helps analyse bad performance or poor resource usage within a program.6

Research

A lot of research has been done within software testing, mainly to do with automation and tools associated with it. The most recent project would be RE-COST - REducing the Cost of Oracles for Software Testing. As explained before, an oracle is "a mechanism for determining whether a test has passed or failed"7. Determining whether the correct behaviour is achieved that may be simple for humans but harder for the software to decide - this is called the oracle problem.8 What RE-COST will do is develop Search Based Optimisation techniques to solve the Oracle Cost problem. It aims to reduce the number of test cases and evaluation effort and reduce the cognition time.

Currently, most of the testing done is done manually as the software may incorrectly identify some behaviour of the program. However, it requires a lot of time and effort for humans to get it right. Therefore, it is essential that automation of some of these tasks are to be carried out as this can drastically improve reliability. Researchers have been making tools to extract as many realistic test cases from web services and use such cases to create a 'one size fits all' fault localisation algorithm.

The whole point of RE-COST is to have the most widespread impact on software testing. So it will likely affect up to 50% of people employed in software testing, as well as improving the software applications throughout households, businesses and industries. Not only will it be affecting the quality of software on desktops or other more visible and more common devices but it will also affect transportation, health, military devices and building infrastructure globally.9

Researchers on RE-COST include Prof Harman, Dr Clark and Dr McMinn with Berner and Mattner, Motorola and Sogeti UK Ltd as project partners.
